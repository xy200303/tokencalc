# tokencalc 技术开发文档

## 1. 项目信息

- 仓库地址: `https://github.com/xy200303/tokencalc`
- 项目语言: Go
- 项目定位: 独立的 token 统计与估算库
- 当前状态: 截至 2026-05-22，远程仓库为空仓库，尚无提交记录

`tokencalc` 的目标不是做网关、计费系统或代理服务，而是沉淀一个纯计算能力库，专门解决不同大模型协议下的 token 统计、估算、归一化和可解释输出问题。

## 2. 目标与边界

### 2.1 核心目标

第一版聚焦以下能力：

1. 根据模型名解析 tokenizer 或 encoding。
2. 解析不同协议的请求体与响应体。
3. 统计或估算 `prompt_tokens`、`completion_tokens`、`total_tokens`。
4. 兼容上游已返回 usage 和上游未返回 usage 两种情况。
5. 输出估算来源、编码方式、支持状态、说明信息。
6. 提供流式响应的增量收集能力。

### 2.2 非目标

以下内容不属于 `tokencalc` 的职责范围：

- 金额计算
- 钱包冻结、结算、扣费
- Redis、数据库、消息队列接入
- HTTP 转发或 provider 路由
- 业务侧鉴权
- usage 落库

项目边界要尽量收紧，避免把库做成“网关计费中台”。

## 3. V1 支持范围

建议第一版支持以下协议与能力：

- `openai_chat`
- `openai_responses`
- `anthropic_messages`
- `gemini_contents`
- 文本 token 计数
- 上游 usage 归一化
- 本地兜底估算
- 流式输出收集
- 多模态占位符策略

V1 不追求所有 provider 的 100% 精确对齐，而是优先做到“稳定、可解释、易扩展”。

## 4. 总体设计

整体流程建议统一为：

1. 解析调用参数，识别协议和模型。
2. 通过模型映射表解析对应 encoding。
3. 优先读取上游返回的 usage。
4. 当 usage 缺失、为空或不完整时，走本地估算。
5. 协议解析器提取 prompt 文本和 completion 文本。
6. tokenizer 对文本做 token 计数。
7. 合并 `reported usage` 与 `local estimate`。
8. 对结果做归一化修正。
9. 输出统一结果和元信息。

建议把“协议解析”和“token 计数”明确分层，避免各协议实现里直接掺杂编码逻辑。

## 6. 核心数据结构

建议先把 API 稳定下来，再开始实现细节。可以参考如下接口：

```go
package tokencalc

type Protocol string

const (
	ProtocolOpenAIChat      Protocol = "openai_chat"
	ProtocolOpenAIResponses Protocol = "openai_responses"
	ProtocolAnthropic       Protocol = "anthropic_messages"
	ProtocolGemini          Protocol = "gemini_contents"
)

type Usage struct {
	PromptTokens     int `json:"prompt_tokens"`
	CompletionTokens int `json:"completion_tokens"`
	TotalTokens      int `json:"total_tokens"`
}

type EstimateSource string

const (
	SourceReportedUsage EstimateSource = "reported_usage"
	SourceLocalEstimate EstimateSource = "local_estimate"
	SourceMerged        EstimateSource = "merged"
	SourceUnsupported   EstimateSource = "unsupported"
)

type EstimateRequest struct {
	Protocol      Protocol
	RequestModel  string
	UpstreamModel string
	IsStream      bool
	RequestBody   []byte
	ResponseBody  []byte
	ReportedUsage *Usage
}

type EstimateResult struct {
	Usage              Usage
	Source             EstimateSource
	Encoding           string
	Supported          bool
	Note               string
	PromptTextLen      int
	CompletionTextLen  int
}

type Service interface {
	Estimate(req EstimateRequest) (EstimateResult, error)
	CountText(model string, text string) (count int, encoding string, err error)
}
```

如果后续需要可配置的图片、音频、文件占位策略，可以继续补一层配置对象：

```go
type PlaceholderPolicy struct {
	ImageTokenCost int
	AudioTokenCost int
	FileTokenCost  int
}
```

## 7. 关键模块职责

### 7.1 `Service`

对外统一入口，负责：

- 路由协议实现
- 选择 tokenizer
- 控制 usage 优先级
- 合并与归一化结果
- 返回统一结果

### 7.2 `Provider Estimator`

每种协议一个实现，职责只放在“从 JSON 中提取文本和结构信息”，不要直接耦合计费逻辑。

建议内部接口类似：

```go
type ProviderEstimator interface {
	ExtractPrompt(body []byte) (text string, extraTokens int, supported bool, note string, err error)
	ExtractCompletion(body []byte, isStream bool) (text string, supported bool, note string, err error)
}
```

这里的 `extraTokens` 用于处理协议固定开销或占位符补偿。

### 7.3 `Codec`

负责把文本转成 token 数，不关心请求协议。

建议能力：

- 根据 encoding 创建编码器
- 统计文本 token 数
- 对未知模型回退默认 encoding

### 7.4 `Model Resolver`

负责从模型名解析到 encoding，建议两层策略：

1. 精确匹配
2. 前缀匹配 fallback

例如：

- `gpt-5`、`gpt-4o`、`o1`、`o3`、`o4`、`qwen` 可映射到 `o200k_base`
- 较老模型可映射到 `cl100k_base` 或其它兼容 encoding

最终映射要做成表驱动，便于后续增补。

### 7.5 `Stream Collector`

流式场景单独抽象，避免把 SSE 拼接逻辑写死在协议估算器中。

建议接口：

```go
type StreamCollector interface {
	AddChunk(part []byte) error
	FinalBody() []byte
}
```

网关或调用方只负责不断把分片喂进来，`tokencalc` 负责产出最终可估算的聚合结果。

## 8. usage 处理策略

### 8.1 优先级规则

第一版建议统一采用以下优先级：

1. 如果上游 usage 完整且有效，优先使用上游 usage。
2. 如果上游 usage 部分缺失，则补齐缺失字段。
3. 如果上游 usage 为 `null`、空对象或全 0，则使用本地估算。
4. 如果协议暂不支持解析，则返回 `unsupported` 并附说明。

### 8.2 归一化规则

建议实现一个明确的 `NormalizeUsage`：

- 只有 `total_tokens` 时，保留 total，其余字段允许为 0。
- 只有 `prompt_tokens` 时，`total_tokens` 至少等于 prompt。
- 只有 `completion_tokens` 时，`total_tokens` 至少等于 completion。
- 同时存在 prompt 和 completion，但 total 为 0 时，自动补成两者之和。
- 如果 `total_tokens < prompt_tokens + completion_tokens`，修正为两者之和。

这部分能力很关键，因为很多上游返回的 usage 格式并不稳定。

## 9. 协议解析策略

### 9.1 OpenAI Chat

重点处理：

- `messages`
- `system` / `user` / `assistant` / `tool`
- 文本内容
- `tool_calls`
- `function_call`
- 响应中的 `choices[].message`
- 流式 `delta`

### 9.2 OpenAI Responses

重点处理：

- `input`
- `output`
- 多段内容数组
- tool 输出片段
- 响应式结构中的文本聚合

### 9.3 Anthropic Messages

重点处理：

- `messages`
- `content` 数组
- `text` / `tool_use` / `tool_result`
- 流式事件中的增量文本

### 9.4 Gemini Contents

重点处理：

- `contents`
- `parts`
- `text`
- 非文本 part 的占位策略
- `candidates` 输出文本聚合

## 10. 多模态处理策略

第一版建议采用“占位符策略”，而不是追求图片、音频、文件的完全精确计数。

建议规则：

- 文本内容按 tokenizer 正常统计
- 图片内容按固定权重计数
- 音频内容按固定权重计数
- 文件内容按固定权重计数
- 结果中通过 `Note` 明确说明存在占位估算

这样做的优点是：

- 实现简单
- 结果稳定
- 便于后续替换为更精细的 provider 特定规则

## 11. 错误处理与可解释性

建议对外暴露的结果不要只有 token 数，还要有足够的排障信息。

建议至少返回：

- `encoding`
- `source`
- `supported`
- `note`
- `prompt_text_len`
- `completion_text_len`

典型说明文案示例：

- `reported usage used directly`
- `reported usage incomplete, merged with local estimate`
- `response body missing, only prompt estimated`
- `unknown model, fallback encoding applied`
- `non-text parts counted by placeholder policy`

## 12. 测试策略

建议从第一天开始写表驱动单测，不要等功能做完再补。

至少覆盖以下场景：

1. OpenAI Chat 非流式请求与响应。
2. OpenAI Chat 流式响应聚合。
3. OpenAI Responses 基本场景。
4. Anthropic messages 文本输入输出。
5. Gemini contents 文本输入输出。
6. `usage: null`。
7. `usage: {}`。
8. `usage` 全 0。
9. 只有 `total_tokens`。
10. 只有 `prompt_tokens`。
11. `total_tokens < prompt + completion`。
12. 未知模型走 fallback encoding。
13. 请求体非法 JSON。
14. 响应体为空。
15. tool calls / function calls。
16. 图片或文件占位策略。
17. `reported usage` 与本地估算合并。
18. 协议不支持时返回 `unsupported`。

额外建议：

- 使用 `testdata/` 保存真实样本
- 用表驱动测试覆盖不同模型映射
- 增加 benchmark 评估大请求体性能

## 13. 开发阶段规划

### 阶段一：项目初始化

交付目标：

- 初始化 `go.mod`
- 建立根包与目录结构
- 定义 `Usage`、`EstimateRequest`、`EstimateResult`
- 定义 `Service` 接口
- 搭建基础单测框架

### 阶段二：核心能力

交付目标：

- 实现 model resolver
- 实现 codec 抽象
- 实现 `CountText`
- 实现 `NormalizeUsage`

### 阶段三：优先协议落地

交付目标：

- 实现 `openai_chat`
- 实现 `openai_responses`
- 打通 `Estimate`
- 补齐对应 `testdata`

这是最值得优先完成的一阶段，因为它最容易尽快形成可用版本。

## 14. 首版完成标准

满足以下条件即可认为 `v0.1.0` 可发布：

1. `CountText` 可稳定工作。
2. `Estimate` 至少支持四类协议中的文本主路径。
3. `reported usage` 与本地估算可正确合并。
4. 流式场景可收敛出可估算的最终响应体。
5. 单测覆盖关键边界场景。
6. README 提供最小使用示例。
7. 对不支持的输入能返回清晰说明，而不是静默给错结果。

## 15. 后续扩展方向

第一版稳定后，可以继续考虑：

- 更多 provider 协议适配
- provider 级精细计数规则
- 更准确的多模态 token 估算
- 暴露流式中间统计结果
- 插件化 provider 注册机制
- 更细粒度的 usage 差异分析

## 16. 建议的首个开发任务

如果要开始落地，建议按下面的顺序推进：

1. 初始化仓库与 `go.mod`。
2. 写 `types.go`、`service.go`、`normalize.go`。
3. 补 `CountText` 和 model resolver。
4. 先完成 `openai_chat`。
5. 再完成 `openai_responses`。
6. 最后补 `anthropic`、`gemini` 和流式支持。

这个顺序的好处是可以尽快得到一个可测、可用、可迭代的最小版本。
